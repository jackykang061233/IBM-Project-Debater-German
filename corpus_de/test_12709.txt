Das Dynamic Host Configuration Protocol (DHCP) ist ein Netzwerk-Management-Protokoll, das in Internet Protocol (IP)-Netzwerken verwendet wird, um IP-Adressen und andere Kommunikationsparameter automatisch an Geräte zuzuordnen, die mit einer Client-Server-Architektur an das Netzwerk angeschlossen sind. Die Technologie eliminiert die Notwendigkeit einer individuellen Konfiguration von Netzwerkgeräten manuell und besteht aus zwei Netzwerkkomponenten, einem zentral installierten Netzwerk-DHCP-Server und Client-Instanzen des Protokollstapels auf jedem Computer oder Gerät. In Verbindung mit dem Netzwerk und periodisch danach fordert ein Client eine Reihe von Parametern aus dem DHCP-Server mit dem DHCP-Protokoll an. DHCP kann in Netzwerken von Wohnnetzen bis zu großen Campus-Netzwerken und regionalen ISP-Netzen implementiert werden. Viele Router und Wohn-Gateways haben DHCP-Serverfähigkeit. Die meisten Reiserouter erhalten im ISP-Netzwerk eine einzigartige IP-Adresse. Innerhalb eines lokalen Netzwerks gibt ein DHCP-Server jedem Gerät eine lokale IP-Adresse zu. DHCP-Dienste existieren für Netzwerke mit Internet Protocol Version 4 (IPv4,) sowie Version 6 (IPv6). Die IPv6-Version des DHCP-Protokolls wird häufig DHCPv6 genannt. HistoryThe Reverse Address Resolution Protocol (RARP) wurde 1984 in RFC 903 für die Konfiguration einfacher Geräte, wie scheibenlose Workstations, mit einer geeigneten IP-Adresse definiert. In der Daten-Link-Schicht war die Implementierung in vielen Server-Plattformen schwierig. Es muss ein Server auf jedem einzelnen Netzwerklink vorhanden sein. RARP wurde durch das im September 1985 in RFC 951 definierte Bootstrap Protocol (BOOTP) ersetzt. Dies führte das Konzept eines Relais-Agents ein, der die Weiterleitung von BOOTP-Paketen über Netzwerke erlaubte, so dass ein zentraler BOOTP-Server Hosts auf vielen IP-Subnetzen bedienen kann. DHCP basiert auf BOOTP, kann aber IP-Adressen aus einem Pool dynamisch zuordnen und wieder zurückfordern, wenn sie nicht mehr im Einsatz sind. Es kann auch verwendet werden, um eine breite Palette von zusätzlichen Konfigurationsparametern an IP-Clients zu liefern, einschließlich plattformspezifischer Parameter. DHCP wurde zunächst in RFC 1531 im Oktober 1993 definiert, aber aufgrund von Fehlern im redaktionellen Prozess wurde fast sofort wieder als RFC 1541 veröffentlicht. Vier Jahre später wurde der DHCPINFORM-Nachrichtentyp und andere kleine Änderungen von RFC 2131 hinzugefügt, die ab 2014 der Standard für IPv4-Netzwerke bleibt. DHCPv6 wurde 2003 von RFC 3315 zunächst beschrieben, dies wurde jedoch von vielen nachfolgenden RFCs aktualisiert. RFC 3633 fügte einen DHCPv6-Mechanismus für die Prefix-Delegation hinzu, und die Stateless-Adresse Autoconfiguration wurde von RFC 3736 hinzugefügt. Überblick Internet Protocol (IP) definiert, wie Geräte innerhalb und über lokale Netzwerke im Internet kommunizieren. Ein DHCP-Server kann IP-Einstellungen für Geräte in seinem lokalen Netzwerk verwalten, z.B. indem IP-Adressen diesen Geräten automatisch und dynamisch zugewiesen werden. DHCP arbeitet basierend auf dem Client-Server-Modell. Wenn ein Computer oder ein anderes Gerät mit einem Netzwerk verbunden ist, sendet die DHCP-Client-Software eine DHCP-Übertragung, die die erforderlichen Informationen anfordert. Jeder DHCP-Server im Netzwerk kann die Anfrage bedienen. Der DHCP-Server verwaltet einen Pool von IP-Adressen und Informationen über Client-Konfigurationsparameter wie Standard-Gateway, Domainname, Nameserver und Zeitserver. Beim Empfang einer DHCP-Anforderung kann der DHCP-Server mit bestimmten Informationen für jeden Client, wie zuvor von einem Administrator konfiguriert, oder mit einer bestimmten Adresse und anderen für das gesamte Netzwerk gültigen Informationen und für den Zeitraum, für den die Zuordnung (Bitte) gültig ist, reagieren. Ein DHCP-Client sucht diese Informationen typischerweise unmittelbar nach dem Booten und periodisch danach vor dem Ablauf der Informationen. Wenn ein DHCP-Client eine Zuordnung erfrischt, fordert er zunächst die gleichen Parameterwerte an, aber der DHCP-Server kann basierend auf den von Administratoren gesetzten Zuordnungsrichtlinien eine neue Adresse zuordnen. Auf großen Netzwerken, die aus mehreren Links bestehen, kann ein einzelner DHCP-Server das gesamte Netzwerk bedienen, wenn es von DHCP-Relaismitteln unterstützt wird, die sich auf den Verbindungsroutern befinden. Solche Agenten übertragen Nachrichten zwischen DHCP-Clients und DHCP-Servern auf verschiedenen Subnetzen. Je nach Implementierung kann der DHCP-Server drei Methoden zur Zuordnung von IP-Adressen haben: Dynamische Zuordnung Ein Netzwerkadministrator behält sich eine Reihe von IP-Adressen für DHCP vor, und jeder DHCP-Client im LAN ist konfiguriert, um eine IP-Adresse vom DHCP-Server bei der Netzwerk-Initialisierung anzufordern. Der Request-and-grant-Prozess verwendet ein Leasingkonzept mit einer steuerbaren Zeitdauer, so dass der DHCP-Server zurückgewonnen werden und dann IP-Adressen, die nicht erneuert werden, zu implementieren.Automatische Zuordnung Der DHCP-Server ordnet einem anfragenden Client dauerhaft eine IP-Adresse aus einem von einem Administrator definierten Bereich zu. Dies ist wie dynamische Zuordnung, aber der DHCP-Server hält eine Tabelle der früheren IP-Adressenzuordnungen, so dass er bevorzugt einem Client die gleiche IP-Adresse zuweisen kann, die der Client zuvor hatte. Manuelle Zuordnung Dieses Verfahren wird auch als statische DHCP-Zuweisung, feste Adresszuweisung, Reservierung und MAC/IP-Adressenbindung bezeichnet. Ein Administrator erstellt für jeden Client eine eindeutige Kennung (eine Client-ID oder MAC-Adresse) an eine IP-Adresse, die dem anfragenden Client angeboten wird. DHCP-Server können konfiguriert werden, um auf andere Methoden zurückfallen, wenn dies ausfällt. DHCP-Dienste werden für Internet Protocol Version 4 (IPv4) und IPv6 verwendet. Die Einzelheiten des Protokolls für IPv4 und IPv6 unterscheiden sich so, dass sie als separate Protokolle betrachtet werden können. Für den IPv6-Betrieb können Geräte alternativ eine staatenlose Adressenautokonfiguration verwenden. IPv6-Hosts können auch die link-lokale Adressierung verwenden, um Operationen zu erreichen, die auf die lokale Netzwerkverbindung beschränkt sind. Betrieb Das DHCP verwendet ein verbindungsloses Servicemodell mit dem User Datagram Protocol (UDP). Es wird mit zwei UDP-Portnummern für seine Vorgänge implementiert, die die gleichen sind wie für das Bootstrap-Protokoll (BOOTP). UDP-Portnummer 67 ist der Zielport eines Servers, und UDP-Portnummer 68 wird vom Client verwendet. DHCP-Betriebe fallen in vier Phasen: Server-Erkennung, IP-Leasing-Angebot, IP-Leasing-Anforderung und IP-Leasing-Erkennung. Diese Stufen sind oft als DORA für Entdeckung, Angebot, Anfrage und Anerkennung abgekürzt. Die DHCP-Operation beginnt mit Clients, die eine Anfrage senden. Sind Client und Server auf verschiedenen Subnetzen, kann ein DHCP Helper oder DHCP Relay Agent verwendet werden. Kunden, die eine Erneuerung eines bestehenden Mietvertrags beantragen, können direkt über UDP unicast kommunizieren, da der Kunde bereits an diesem Punkt eine etablierte IP-Adresse hat. Zusätzlich gibt es ein BROADCAST-Flag (1 Bit in 2 Byte-Flags-Feld, wo alle anderen Bits reserviert sind und so auf 0 gesetzt werden) kann der Client verwenden, um anzuzeigen, auf welche Weise (Broadcast oder Unicast) es den DHCPOFFER:0x8000 für die Sendung empfangen kann, 0x0000 für unicast. Normalerweise wird der DHCPOFFER durch Unicast gesendet. Für diejenigen Hosts, die Unicast-Pakete nicht akzeptieren können, bevor IP-Adressen konfiguriert sind, kann dieses Flag verwendet werden, um dieses Problem zu bearbeiten. Entdeckung Der DHCP-Client sendet eine DHCPDISCOVER-Nachricht im Netzteilnetz mit der Zieladresse 255.255.255.255 (begrenzte Sendung) oder der spezifischen Subnetz-Sendeadresse (direkte Sendung). A DHCP-Client kann auch seine letzte bekannte IP-Adresse anfordern. Wenn der Client mit demselben Netzwerk verbunden bleibt, kann der Server die Anfrage erteilen. Andernfalls hängt es davon ab, ob der Server als autoritativ eingerichtet ist oder nicht. Ein maßgeblicher Server verwehrt die Anfrage, wodurch der Client eine neue Anfrage ausgibt. Ein nicht autoritativer Server ignoriert einfach die Anfrage, was zu einem implementierungsabhängigen Timeout für den Client führt, um die Anfrage abzubrechen und eine neue IP-Adresse zu verlangen. Wenn beispielsweise HTYPE auf 1 gesetzt wird, um anzugeben, dass das verwendete Medium Ethernet ist, wird HLEN auf 6 gesetzt, weil eine Ethernet-Adresse (MAC-Adresse) 6 Oktette lang ist. Der CHADDR wird auf die vom Client verwendete MAC-Adresse eingestellt. Einige Optionen sind ebenfalls festgelegt. Angebot Wenn ein DHCP-Server eine DHCPDISCOVER-Nachricht von einem Client erhält, der eine IP-Adresse-Leasing-Anforderung ist, behält sich der DHCP-Server eine IP-Adresse für den Client vor und macht ein Leasingangebot durch Senden einer DHCPOFFER-Nachricht an den Client. Diese Nachricht enthält die Client-ID des Clients (traditionell eine MAC-Adresse), die IP-Adresse, die der Server anbietet, die Subnet-Maske, die Leasingdauer und die IP-Adresse des DHCP-Servers, der das Angebot macht. Der DHCP-Server kann auch die Hardware-Level-MAC-Adresse in der darunterliegenden Transportschicht bemerken: Gemäß aktuellen RFCs kann die Transportschicht MAC-Adresse verwendet werden, wenn keine Client-ID im DHCP-Paket bereitgestellt wird. Der DHCP-Server ermittelt die Konfiguration basierend auf der Hardwareadresse des Clients, wie sie im Feld CHADDR (Client Hardware Address) angegeben ist. Hier gibt der Server 192.168.1.1 die IP-Adresse des Clients im Feld YIADDR (Ihre IP-Adresse) an. Anfrage In Reaktion auf das DHCP-Angebot antwortet der Client mit einer DHCPREQUEST-Nachricht, sendet an den Server und fordert die angebotene Adresse. Ein Client kann DHCP-Angebote von mehreren Servern empfangen, aber es wird nur ein DHCP-Angebot akzeptieren. Der Client wird eine unentgeltliche ARP produzieren, um zu finden, ob es einen anderen Host im Netzwerk mit derselben IP-Adresse gibt.Gibt es keine Antwort von einem anderen Host, dann gibt es keinen Host mit derselben IP-Konfiguration im Netzwerk und die Nachricht wird an den Server gesendet, der die Akzeptanz der IP-Adresse zeigt. Der Client muss die Serveridentifikationsoption in der Anforderungsnachricht senden, die den Server angibt, dessen Angebot der Client ausgewählt hat. Wenn andere DHCP-Server diese Nachricht erhalten, widerrufen sie alle Angebote, die sie an den Client gemacht haben und die angebotene IP-Adresse an den Pool der verfügbaren Adressen zurückgeben. Anerkennung Wenn der DHCP-Server die DHCPREQUEST-Nachricht vom Client erhält, gibt der Konfigurationsprozess seine endgültige Phase ein. Die Bestätigungsphase beinhaltet das Senden eines DHCPACK-Pakets an den Client. Dieses Paket enthält die Mietdauer und alle anderen Konfigurationsinformationen, die der Client angefordert haben könnte. An dieser Stelle wird der IP-Konfigurationsprozess abgeschlossen. Das Protokoll erwartet, dass der DHCP-Client seine Netzwerkschnittstelle mit den ausgehandelten Parametern konfiguriert. Nachdem der Client eine IP-Adresse erhält, sollte er die neu empfangene Adresse (z.B. mit ARP Address Resolution Protocol) abtasten, um Adresskonflikte zu verhindern, die durch überlappende Adresspools von DHCP-Servern verursacht werden. Relais In kleinen Netzwerken, in denen nur ein IP-Subnetz verwaltet wird, kommunizieren DHCP-Clients direkt mit DHCP-Servern. DHCP-Server können jedoch auch IP-Adressen für mehrere Subnetze bereitstellen. In diesem Fall kann ein DHCP-Client, der noch keine IP-Adresse erworben hat, nicht direkt mit dem DHCP-Server über IP-Routing kommunizieren, da er keine routable IP-Adresse hat, die Link-Layer-Adresse eines Routers nicht kennt und die IP-Adresse des DHCP-Servers nicht kennt. Um DHCP-Clients auf Subnetzen zu ermöglichen, die nicht direkt von DHCP-Servern zur Kommunikation mit DHCP-Servern dienen, können DHCP-Relais Agenten auf diesen Subnetzen installiert werden.Der DHCP-Client sendet auf dem lokalen Link; der Relaisagent empfängt die Sendung und sendet sie an einen oder mehrere DHCP-Server mit unicast. Der Relais Agent speichert seine eigene IP-Adresse im Feld GIADDR Feld des DHCP-Pakets. Der DHCP-Server verwendet den GIADDR-Wert, um das Subnetz, auf dem der Relaisagent die Sendung empfangen hat, zu bestimmen und eine IP-Adresse auf diesem Subnetz zuzuordnen. Wenn der DHCP-Server an den Client antwortet, sendet er die Antwort auf die GIADDR-Adresse, erneut mit Unicast. Der Relaisagent überträgt dann die Antwort auf das lokale Netz. In dieser Situation verwendet die Kommunikation zwischen dem Relais-Agent und dem DHCP-Server typischerweise sowohl einen Source- als auch Ziel-UDP-Port von 67. Zuverlässigkeit Das DHCP sorgt für Zuverlässigkeit auf verschiedene Weise: periodische Erneuerung, Wiederverbindlichkeit und Ausfall. DHCP-Kunden werden Leasingverträge zugewiesen, die für einige Zeit dauern. Kunden beginnen zu versuchen, ihre Mieten zu erneuern, sobald die Hälfte des Mietintervalls abgelaufen ist. Sie tun dies, indem sie eine unicast DHCPREQUEST-Nachricht an den DHCP-Server senden, der den ursprünglichen Mietvertrag erteilt hat. Wenn dieser Server nicht oder nicht erreichbar ist, wird er nicht auf den DHCPREQUEST reagieren. In diesem Fall wiederholt der Client jedoch den DHCPREQUEST von Zeit zu Zeit, so dass, wenn der DHCP-Server wieder zurückkommt oder wieder erreichbar wird, der DHCP-Client gelingt, ihn zu kontaktieren und den Mietvertrag zu verlängern. Wenn der DHCP-Server für einen längeren Zeitraum nicht erreichbar ist, wird der DHCP-Client versuchen, zu rebinieren, indem er seine DHCPREQUEST sendet, anstatt sie zu unicasting. Da es ausgestrahlt wird, wird die DHCPREQUEST-Nachricht alle verfügbaren DHCP-Server erreichen. Wenn ein anderer DHCP-Server in der Lage ist, den Mietvertrag zu verlängern, wird dies zu diesem Zeitpunkt tun. Um wieder zu arbeiten, wenn der Client erfolgreich einen Backup-DHCP-Server kontaktiert, muss dieser Server genaue Informationen über die Bindung des Clients haben. Die Aufrechterhaltung der genauen Bindungsinformationen zwischen zwei Servern ist ein kompliziertes Problem; wenn beide Server die gleiche Leasing-Datenbank aktualisieren können, muss es einen Mechanismus geben, um Konflikte zwischen Updates auf den unabhängigen Servern zu vermeiden. Ein Vorschlag zur Umsetzung von fehlertoleranten DHCP-Servern wurde der Internet Engineering Task Force vorgelegt, aber nie formalisiert. Wenn die Wiederverbindlichkeit ausfällt, läuft der Mietvertrag schließlich ab. Wenn der Mietvertrag abläuft, muss der Kunde die ihm in seinem Mietvertrag gewährte IP-Adresse nicht verwenden. Zu diesem Zeitpunkt wird der DHCP-Prozess von Anfang an neu gestartet, indem er eine DHCPDISCOVER-Nachricht aussendet. Da der Mietvertrag abgelaufen ist, wird er die ihm angebotene IP-Adresse akzeptieren. Sobald es eine neue IP-Adresse (vermutlich von einem anderen DHCP-Server) hat, wird es wieder in der Lage sein, das Netzwerk zu nutzen. Da sich jedoch seine IP-Adresse geändert hat, werden laufende Verbindungen gebrochen. Diese Angriffe fallen in drei Hauptkategorien: Unautorisierte DHCP-Server, die den Clients falsche Informationen liefern. Unbefugte Kunden erhalten Zugriff auf Ressourcen. Resource Erschöpfung Angriffe von bösartigen DHCP-Clients. Da der Client keine Möglichkeit hat, die Identität eines DHCP-Servers zu validieren, können nicht autorisierte DHCP-Server (gemeinsam "rogue DHCP" genannt) in Netzwerken betrieben werden und DHCP-Clients fehlerhafte Informationen bereitstellen. Dies kann entweder als Denial-of-Service-Angriff dienen, um zu verhindern, dass der Client Zugriff auf Netzwerk-Konnektivität erlangt, oder als Man-in-the-Middle-Angriff. Da der DHCP-Server dem DHCP-Client Server mit Server-IP-Adressen wie der IP-Adresse eines oder mehrerer DNS-Server zur Verfügung stellt, kann ein Angreifer einen DHCP-Client davon überzeugen, seine DNS-Lookups über einen eigenen DNS-Server zu erledigen und eigene Antworten auf DNS-Abfragen des Clients bereitzustellen. Dies wiederum ermöglicht es dem Angreifer, den Netzwerkverkehr durch sich selbst umzuleiten, so dass er auf Verbindungen zwischen dem Client und den Netzwerkservern, die er kontaktiert, eavesdrop oder einfach diese Netzwerkserver mit eigenen ersetzen kann.Da der DHCP-Server keinen sicheren Mechanismus zur Authentisierung des Clients hat, können Clients einen unbefugten Zugriff auf IP-Adressen gewinnen, indem sie Anmeldeinformationen, wie z.B. Client-Identifikationen, präsentieren, die anderen DHCP-Clients gehören. Dadurch können DHCP-Clients auch den Speicher der IP-Adressen des DHCP-Servers ausschöpfen – indem er jedes Mal, wenn er eine Adresse fordert, neue Anmeldeinformationen vorlegt, kann der Client alle verfügbaren IP-Adressen auf einem bestimmten Netzwerk-Link verbrauchen und andere DHCP-Clients daran hindern, Dienst zu bekommen. DHCP bietet einige Mechanismen zur Minderung dieser Probleme. Die Protokollerweiterung Relay Agent Information Option (RFC 3046, in der Regel durch die tatsächliche Nummer als Option 82 bezeichnet) ermöglicht es Netzbetreibern, Tags an DHCP-Nachrichten zu befestigen, da diese Nachrichten im vertrauenswürdigen Netzwerk des Netzbetreibers ankommen. Dieses Tag wird dann als Berechtigungstoken verwendet, um den Zugriff des Clients auf Netzwerkressourcen zu steuern. Da der Client keinen Zugriff auf das Netzwerk vor dem Relay-Agent hat, verhindert die fehlende Authentifizierung nicht, dass der DHCP-Server-Operator sich auf den Berechtigungstoken verlässt. Eine weitere Erweiterung, Authentication für DHCP-Nachrichten (RFC 3118), bietet einen Mechanismus zur Authentifizierung von DHCP-Nachrichten. Seit 2002 war RFC 3118 aufgrund der Probleme der Verwaltung von Schlüsseln für viele DHCP-Kunden nicht weit verbreitet.
