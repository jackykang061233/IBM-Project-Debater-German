Ein Software-Repository oder kurz „Repo“ ist ein Speicherort für Softwarepakete. Oft wird auch eine Inhaltstabelle zusammen mit Metadaten gespeichert. Ein Software-Repository wird typischerweise von Quell- oder Repository-Managern verwaltet. Paketmanager ermöglichen die Installation und Aktualisierung der Repositories (manchmal als "Pakete") und müssen dies manuell tun. Viele Softwareverleger und andere Organisationen pflegen Server im Internet zu diesem Zweck, entweder kostenlos oder für eine Abonnementgebühr. Repositories können ausschließlich für bestimmte Programme wie CPAN für die Perl-Programmiersprache oder für ein gesamtes Betriebssystem sein. Betreiber solcher Repositories stellen typischerweise ein Paketverwaltungssystem zur Verfügung, Werkzeuge, die dazu bestimmt sind, Softwarepakete aus den Repositories zu suchen, zu installieren und anderweitig zu manipulieren. Zum Beispiel verwenden viele Linux-Distributionen Advanced Packaging Tool (APT), die häufig in Debian-basierten Distributionen gefunden werden, oder yum in Red Hat-basierten Distributionen gefunden. Es gibt auch mehrere unabhängige Paketverwaltungssysteme, wie pacman, in Arch Linux und equo verwendet, gefunden in Sabayon Linux. Da Software-Repositorys entwickelt sind, um nützliche Pakete, große Repositories sind entworfen, um Malware frei zu sein. Wenn ein Computer konfiguriert ist, ein digital signiertes Repository von einem seriösen Anbieter zu verwenden und mit einem entsprechenden Berechtigungssystem gekoppelt ist, reduziert dies die Bedrohung von Malware auf diese Systeme erheblich. Als Nebeneffekt benötigen viele Systeme, die diese Fähigkeiten haben, keine Anti-Malware-Software wie Anti-Virus-Software. Die meisten großen Linux-Distributionen haben viele Repositories auf der ganzen Welt, die das Hauptrepository spiegeln. In einer Unternehmensumgebung wird ein Software-Repository in der Regel verwendet, um Artefakte zu speichern oder externe Repositories zu spiegeln, die aufgrund von Sicherheitseinschränkungen unzugänglich sein können. Solche Repositories können zusätzliche Funktionalität bieten, wie Zugriffskontrolle, Versionierung, Sicherheitsüberprüfungen für hochgeladene Software, Cluster-Funktionalität etc. und unterstützen typischerweise eine Vielzahl von Formaten in einem Paket, um alle Bedürfnisse in einem Unternehmen zu erfüllen und so einen einzigen Punkt der Wahrheit zu bieten. Beliebte Beispiele sind JFrog Artifactory und Nexus Repository. Auf Clientseite hilft ein Paketmanager beim Installieren und Aktualisieren der Repositories. Serverseitig wird ein Software-Repository typischerweise von Quell- oder Repository-Managern verwaltet. Einige der Repository-Manager erlauben es, einen anderen Repository-Standort in eine URL zu aggregieren und einen Cache-Proxy bereitzustellen. Beim kontinuierlichen Bau werden viele Artefakte produziert und oft zentral gespeichert, so dass das automatische Löschen der nicht freigegebenen ist wichtig. Paketmanagementsystem vs. Paketentwicklungsprozess Ein Paketmanagementsystem unterscheidet sich von einem Paketentwicklungsprozess. Eine typische Verwendung eines Paketmanagementsystems ist die Integration von Code aus eventuell unterschiedlichen Quellen in eine kohärente eigenständige Bedieneinheit zu erleichtern. So kann ein Paketmanagement-System verwendet werden, um eine Distribution von Linux herzustellen, möglicherweise eine auf eine bestimmte eingeschränkte Anwendung zugeschnittene Distribution. Ein Paket-Entwicklungsprozess hingegen dient dazu, die Ko-Entwicklung von Code und Dokumentation einer Sammlung von Funktionen oder Routinen mit einem gemeinsamen Thema zu verwalten, wodurch ein Paket von Software-Funktionen erzeugt wird, die typischerweise nicht von selbst vollständig und nutzbar sind. Ein guter Paketentwicklungsprozess wird den Benutzern helfen, sich an gute Dokumentations- und Kodierungspraktiken anzupassen und einige Maßstäbe zu integrieren.
