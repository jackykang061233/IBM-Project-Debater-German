Extensible Messaging and Presence Protocol (XMPP, ursprünglich Jabber genannt) ist ein offenes Kommunikationsprotokoll, das für Instant Messaging (IM,) Präsenzinformationen und Kontaktlistenwartung konzipiert ist. Basierend auf XML (Extensible Markup Language) ermöglicht es den zeitnahen Austausch von strukturierten Daten zwischen zwei oder mehr Netzwerkeinheiten. Das Protokoll bietet eine Vielzahl von Anwendungen über das traditionelle IM im breiteren Bereich der Nachrichten-orientierten Middleware hinaus, einschließlich Signalisierung für VoIP, Video, Dateiübertragung, Gaming und andere Anwendungen. Im Gegensatz zu den meisten kommerziellen Instant Messaging-Protokollen wird XMPP in einem offenen Standard in der Anwendungsschicht definiert. Die Architektur des XMPP-Netzwerks ist ähnlich wie E-Mail; jeder kann seinen eigenen XMPP-Server ausführen und es gibt keinen zentralen Master-Server. Dieser offene Systemansatz ermöglicht es Benutzern, mit anderen auf jedem Server mit einem JID-Benutzerkonto zu interoperieren, ähnlich einer E-Mail-Adresse. XMPP Implementierungen können mit jeder Softwarelizenz entwickelt werden und viele Server-, Client- und Bibliotheks-Implementierungen werden als kostenlose und Open-Source-Software vertrieben. Es gibt auch zahlreiche Freeware- und kommerzielle Software-Implementierungen. Ursprünglich von der gleichnamigen Open-Source-Community entwickelt, wurden die Protokolle 2004 als anerkannter Instant Messaging Standard formalisiert und mit neuen Erweiterungen und Features kontinuierlich entwickelt. Verschiedene XMPP-Client-Software sind sowohl auf Desktop- als auch auf mobilen Plattformen und Geräten verfügbar - bis 2003 wurde das Protokoll von über zehn Millionen Menschen weltweit im Netzwerk eingesetzt, so die XMPP Standards Foundation. Protokollmerkmale Dezentralisierung Die XMPP Netzwerkarchitektur erinnert an das Simple Mail Transfer Protocol (SMTP,) ein Client-Server-Modell; Clients sprechen nicht direkt miteinander, da es dezentral ist - jeder kann einen Server ausführen. Durch Design gibt es keinen zentralen autoritären Server, wie es mit Messaging-Diensten wie AIM, WLM, WhatsApp oder Telegram gibt. Einige Verwirrung entsteht oft auf diesem Punkt, da es einen öffentlichen XMPP-Server gibt, der bei jabber.org ausgeführt wird, auf den viele Benutzer abonnieren. Jedoch kann jeder seinen eigenen XMPP-Server auf seiner eigenen Domain ausführen. Adressierung Jeder Benutzer im Netzwerk hat eine einzigartige XMPP-Adresse, genannt JID (aus historischen Gründen werden XMPP-Adressen oft Jabber-IDs genannt). Die JID ist wie eine E-Mail-Adresse mit einem Benutzernamen und einem Domain-Namen (oder IP-Adresse) für den Server, auf dem sich dieser Nutzer befindet, getrennt durch ein unterschreiben (@) - z.B. "alice@example.com": hier ist alice der Benutzername und example.com der Server, mit dem der Benutzer registriert ist. Da sich ein Benutzer an mehreren Standorten anmelden möchte, kann er eine Ressource angeben. Eine Ressource identifiziert einen bestimmten Client, der dem Benutzer gehört (z.B. zu Hause, Arbeit oder Mobil). Dies kann in der JID enthalten sein, indem ein Slash, gefolgt von dem Namen der Ressource. Zum Beispiel könnte die vollständige JID des mobilen Benutzerkontos username@example.com/mobile sein. Jede Ressource kann einen numerischen Wert angegeben haben, der Priorität genannt wird. Nachrichten, die einfach an username@example.com gesendet werden, gehen an den Client mit höchster Priorität, aber diejenigen, die an username@example.com/mobile gesendet werden, gehen nur an den mobilen Client. Die höchste Priorität ist diejenige mit dem größten Zahlenwert. JIDs ohne Benutzername-Teil sind ebenfalls gültig und können für Systemnachrichten und Steuerung von Sondermerkmalen auf dem Server verwendet werden. Auch für diese JIDs bleibt eine Ressource optional. Die Mittel, um Nachrichten basierend auf einer logischen Endpunkt-Identifizierung - die JID, statt durch eine explizite IP-Adresse bieten Möglichkeiten, XMPP als Overlay-Netzwerk-Implementierung auf verschiedenen Unterlay-Netzwerken zu verwenden. XMPP via HTTP Das ursprüngliche und native Transportprotokoll für XMPP ist das Transmission Control Protocol (TCP), das offene XML-Streams über langlebige TCP-Verbindungen verwendet. Alternativ zum TCP-Transport hat die XMPP-Community auch einen HTTP-Transport für Web-Clients sowie Benutzer hinter eingeschränkten Firewalls entwickelt. In der ursprünglichen Spezifikation könnte XMPP HTTP auf zwei Arten verwenden: Abfrage und Bindung. Die Abfragemethode, die jetzt depreciert wird, im Wesentlichen impliziert, dass auf einer serverseitigen Datenbank gespeicherte Nachrichten regelmäßig von einem XMPP-Client über HTTP GET- und POST-Anfragen abgerufen (und veröffentlicht) werden. Die mit Bidirektional-streams Over Synchronous HTTP (BOSH) implementierte Bindungsmethode ermöglicht es Servern, Nachrichten an Clients zu schieben, sobald sie gesendet werden. Dieses Push-Modell der Benachrichtigung ist effizienter als die Abfrage, wo viele der Umfragen keine neuen Daten zurückgeben.Da der Client HTTP verwendet, ermöglichen die meisten Firewalls es Clients, Nachrichten ohne Behinderungen zu holen und zu posten. In Szenarien, in denen der von XMPP verwendete TCP-Port gesperrt ist, kann ein Server auf dem normalen HTTP-Port hören und der Verkehr sollte problemlos passieren. Verschiedene Webseiten lassen die Menschen über einen Browser in XMPP anmelden. Darüber hinaus gibt es offene öffentliche Server, die auf Standard-http (port 80) und https (port 443)-Ports hören und somit Verbindungen von hinter den meisten Firewalls zulassen. Allerdings ist der IANA-registrierte Hafen für BOSH tatsächlich 5280, nicht 80. Erweiterbarkeit Die XMPP Standards Foundation oder XSF (früher die Jabber Software Foundation) ist aktiv bei der Entwicklung offener XMPP Erweiterungen, so genannte XEP. Allerdings können Erweiterungen auch durch jedes einzelne, Software-Projekt oder Organisation definiert werden. Zur Aufrechterhaltung der Interoperabilität werden gemeinsame Erweiterungen vom XSF verwaltet. XMPP-Anwendungen außerhalb von IM umfassen: Chaträume, Netzwerkmanagement, Content-Syndion, Collaboration-Tools, Dateifreigabe, Gaming, Remote-Systeme Steuerung und Überwachung, Geolokation, Middleware und Cloud Computing, VoIP und Identitätsdienste. XMPP ist aufgrund seiner Fähigkeit, die Entdeckung in lokalen Netzwerk-Domains zu unterstützen, gut für Cloud-Computing geeignet, wo virtuelle Maschinen, Netzwerke und Firewalls sonst Hindernisse für alternative Service-Erkennungen und Präsenz-basierte Lösungen darstellen würden. Cloud Computing- und Storage-Systeme verlassen sich auf verschiedene Kommunikationsformen über mehrere Ebenen, einschließlich nicht nur Messaging zwischen Systemen zum Relaiszustand, sondern auch die Migration oder Verteilung größerer Objekte, wie Speicher oder virtuelle Maschinen. Neben der Authentifizierung und dem in-transit-Datenschutz kann XMPP auf unterschiedlichen Ebenen angewendet werden und kann sich als erweitertes Middleware- oder Message-orientiertes Middleware (MOM)-Protokoll ideal erweisen. Aktuelle Einschränkungen Im Moment unterstützt XMPP nicht Quality of Service (QoS;) die sichere Auslieferung von Nachrichten muss auf der XMPP-Schicht aufgebaut werden. Es gibt zwei XEPs, die vorgeschlagen werden, um diese Frage zu behandeln, XEP-0184 Nachrichten-Lieferscheine, die derzeit ein Entwurf Standard ist, und XEP-0333 Chat Markers, die als experimentell betrachtet wird. Da XML Text basiert, hat normal XMPP im Vergleich zu rein binären Lösungen ein höheres Netzwerk-Overhead. Dieses Problem wurde vom experimentellen XEP-0322: Efficient XML Interchange (EXI) Format angesprochen, in dem XML sehr effizient binär, insbesondere im schema-informierten Modus, serialisiert wird. Diese XEP wird derzeit verschoben. In-Band binäre Datenübertragung ist begrenzt. Binäre Daten müssen zuerst Basis64 codiert sein, bevor sie in-Band übertragen werden können. Daher wird jede signifikante Menge binärer Daten (z.B. Dateiübertragungen) am besten aus-of-Band übertragen, indem in-Band-Nachrichten zur Koordination verwendet werden. Das beste Beispiel dafür ist das Jingle XMPP Extension Protocol, XEP-0166. Features Peer-to-Peer-Sitzungen Mit der Erweiterung namens Jingle kann XMPP ein offenes Mittel zur Unterstützung von Maschinen-zu-Maschine- oder Peer-to-Peer-Kommunikation über eine Vielzahl von Netzwerken bereitstellen. Diese Funktion wird hauptsächlich für IP-Telefonie (VoIP) verwendet. Multi-User Chat XMPP unterstützt Konferenzen mit mehreren Benutzern unter Verwendung der Spezifikation Multi-User Chat (MUC) (XEP-0045). Aus der Sicht eines normalen Benutzers ist es vergleichbar mit Internet Relay Chat (IRC.) Sicherheits- und Verschlüsselungs-XMPP-Server können isoliert werden (z.B. auf einem Firmen-Intranet), und eine sichere Authentifizierung (SASL) und Punkt-zu-Punkt-Verschlüsselung (TLS) wurden in die Kern-XMPP-Spezifikationen eingebaut, sowie Off-the-Recordaging Mess (OTR) ist eine Erweiterung von XMPP, die eine Verschlüsselung von Nachrichten und Daten ermöglicht. Es wurde seitdem durch eine bessere Erweiterung ersetzt, Multi-End-zu-Multi-End-Verschlüsselung (OMEMO, XEP-0384) End-to-End-Verschlüsselung zwischen Benutzern. Dadurch ergibt sich ein höheres Sicherheitsniveau, indem alle Daten des Quell-Clients verschlüsselt und wieder am Ziel-Client entschlüsselt werden; der Server-Operator kann die von ihnen übermittelten Daten nicht entschlüsseln. Service Entdeckung Während heute mehrere Service-Entdeckungsprotokolle bestehen (z.B. Zeroconf oder das Service Location Protocol), bietet XMPP eine solide Basis für die Entdeckung von Dienstleistungen, die lokal oder über ein Netzwerk liegen, und die Verfügbarkeit dieser Dienste (über Präsenzinformationen), wie von XEP-0030 DISCO angegeben. Verbindung mit anderen Protokollen Eines der ursprünglichen Designziele der frühen Jabber Open-Source-Community ermöglichte es Benutzern, über eine einzige Client-Anwendung mit mehreren Instant Messaging-Systemen (insbesondere Nicht-XMPP-Systeme) zu verbinden. Dies erfolgte durch Einrichtungen, die Transporte oder Gateways zu anderen Instant Messaging-Protokollen wie ICQ, AIM oder Yahoo Messenger, aber auch zu Protokollen wie SMS, IRC oder E-Mail.Im Gegensatz zu Multi-Protokoll-Clients bietet XMPP diesen Zugriff auf Serverebene, indem er über spezielle Gateway-Dienste, die neben einem XMPP-Server laufen, kommuniziert. Jeder Benutzer kann sich mit einem dieser Gateways registrieren, indem er die Informationen liefert, die benötigt werden, um sich an dieses Netzwerk anzumelden, und kann dann mit Benutzern dieses Netzwerks kommunizieren, als wären sie XMPP-Benutzer. So funktionieren solche Gateways als Client-Proxies (das Gateway authentifiziert im Auftrag des Benutzers im Non-XMPP-Service). Somit kann jeder Client, der XMPP vollständig unterstützt, auf jedes Netzwerk mit einem Gateway ohne zusätzlichen Code im Client zugreifen, und ohne dass der Client direkten Zugriff auf das Internet haben muss. Das Client-Proxy-Modell kann jedoch gegen die Nutzungsbedingungen des verwendeten Protokolls verstoßen (obwohl solche Nutzungsbedingungen in mehreren Ländern nicht gesetzlich durchsetzbar sind) und der Nutzer verpflichtet sich auch, seinen IM-Benutzernamen und sein Passwort an die Drittanbieter-Website zu senden, die den Transport betreibt (die Datenschutz- und Sicherheitsbelange erhöhen kann). Eine andere Art von Gateway ist ein Server-to-Server-Gateway, das eine nicht-XMPP-Server-Einstellung ermöglicht, um mit den in Interdomain-Feederation-Funktionen von XMPP erstellten nativen XMPP-Servern zu verbinden. Solche Server-to-Server-Gateways werden von mehreren Enterprise IM Software-Produkten angeboten, darunter: IBM Lotus Sametime Skype for Business Server (früher Microsoft Lync Server und Microsoft Office Communications Server – OCS) Software XMPP wird von vielen Clients, Servern und Codebibliotheken implementiert. Diese Implementierungen werden unter einer Vielzahl von Softwarelizenzen bereitgestellt. Server Es gibt zahlreiche XMPP Server-Software, einige bekannte sind ejabberd und Prosody. Kunden Eine Vielzahl von XMPP-Client-Software gibt es auf verschiedenen modernen und alten Plattformen, darunter sowohl grafische als auch befehlszeilenbasierte Clients. Nach der XMPP-Website, einige der beliebtesten Software umfassen Gajim (Windows, Linux,) Swift.IM (macOS, Windows, Linux,) Conversations (Android,) Monal (macOS, iOS) und Converse.js (Webbrowser, Linux, Windows, macOS). Weitere Kunden sind: Bombus, ChatSecure, Coccinella, JWChat.org, MCabber, Miranda, Pidgin, Psi, Tkabber, Trillian und Xabber. Bereitstellung und Verteilung Es gibt Tausende von XMPP-Servern weltweit, viele öffentliche sowie private Personen oder Organisationen, die ihre eigenen Server ohne kommerzielle Absicht betreiben. Zahlreiche Webseiten zeigen eine Liste von öffentlichen XMPP-Servern, auf denen sich Nutzer registrieren können (z.B. auf der XMPP.net-Website). Mehrere große öffentliche IM-Dienste nutzen oder verwendet XMPP, darunter LiveJournals "LJ Talk", Nimbuzz und HipChat. Verschiedene Hosting-Dienste, wie DreamHost, ermöglichen Hosting-Kunden, XMPP-Dienste neben traditionellen Web- und E-Mail-Diensten zu wählen. Spezielle XMPP Hosting-Dienste existieren auch in Form von Cloud, so dass Domain-Besitzer nicht direkt ihre eigenen XMPP-Server ausführen müssen, einschließlich Cisco Webex Connect, Chrome.pl, Flosoft.biz, i-pobox.net und hosted.im. XMPP wird auch in Bereitstellungen von Nicht-IM-Diensten verwendet, darunter Smart-Grid-Systeme wie Anforderungs-Antwort-Anwendungen, Nachrichten-orientierte Middleware und als Ersatz für SMS, um Textnachrichten auf vielen Smartphone-Clients bereitzustellen. Nicht-native Bereitstellungen Einige der größten Messaging-Anbieter verwenden oder verwendet wurden, verschiedene Formen von XMPP-basierten Protokollen in ihren Backend-Systemen, ohne diese Tatsache unbedingt an ihre Endbenutzer auszusetzen. Ein Beispiel ist Google, die im August 2005 eingeführt Google Talk, eine Kombination VoIP und IM-System, das XMPP für Instant Messaging und als Basis für ein Sprach- und Dateiübertragungs-Signalisierungsprotokoll namens Jingle verwendet. Der erste Start beinhaltete keine Server-to-Server-Kommunikation; Google aktivierte diese Funktion am 17. Januar 2006. Google hat seitdem Video-Funktionalität zu Google Talk hinzugefügt, auch mit dem Jingle-Protokoll zur Signalisierung. Im Mai 2013, Google angekündigt, XMPP-Kompatibilität würde von Google Talk für Server-zu-Server-Feederation fallen, obwohl es würde Client-to-Server-Unterstützung beibehalten. Im Januar 2008 stellte AOL experimentelle XMPP-Unterstützung für seinen AOL Instant Messenger (AIM)-Service vor, sodass AIM-Nutzer mit XMPP kommunizieren können. Im März 2008 wurde dieser Dienst jedoch eingestellt. Ab Mai 2011 bietet AOL begrenzte XMPP-Unterstützung. Im Februar 2010 eröffnete die Social-Networking-Website Facebook ihre Chat-Funktion für Drittanbieter-Anwendungen über XMPP. Einige Funktionen waren durch XMPP nicht verfügbar, und Unterstützung wurde im April 2014 fallen gelassen. Ähnlich, im Dezember 2011, Microsoft veröffentlicht eine XMPP-Schnittstelle zu seinem Microsoft Messenger-Service.Skype, sein de facto Nachfolger, bot auch begrenzte XMPP-Unterstützung. Apache Wave ist ein weiteres Beispiel. XMPP ist der de facto Standard für privates Chat in Glücksspiel-Plattformen wie Origin, und PlayStation, sowie die jetzt eingestellten Xfire und Raptr. Zwei bemerkenswerte Ausnahmen sind Steam und Xbox LIVE; beide verwenden ihre eigenen proprietären Messaging Protokolle. Geschichte und Entwicklung Jeremie Miller begann 1998 mit der Arbeit an der Jabber-Technologie und veröffentlichte die erste Version des Jabberd-Servers am 4. Januar 1999. Die frühe Jabber-Community konzentrierte sich auf Open-Source-Software, vor allem auf den Jabberd-Server, aber sein Hauptergebnis erwies sich als die Entwicklung des XMPP-Protokolls. Die Internet Engineering Task Force (IETF) bildete 2002 eine XMPP-Arbeitsgruppe, um die Kernprotokolle als IETF Instant Messaging und Präsenztechnologie zu formalisieren. Das frühe Jabber-Protokoll, das 1999 und 2000 entwickelt wurde, bildete die Grundlage für XMPP, wie es im Oktober 2004 in RFC 3920 und RFC 3921 veröffentlicht wurde (die primären Änderungen bei der Formalisierung durch die XMPP-Arbeitsgruppe des IETF waren die Hinzufügung von TLS zur Kanalverschlüsselung und SASL zur Authentifizierung). Die Arbeitsgruppe XMPP hat auch Spezifikationen RFC 3922 und RFC 3923 erstellt. 2011 wurden RFC 3920 und RFC 3921 durch RFC 6120 bzw. RFC 6121 ersetzt, wobei RFC 6122 das XMPP-Adressenformat angibt. 2015 wurde RFC 6122 von RFC 7622 überholt. Neben diesen auf der IETF standardisierten Kernprotokollen ist die XMPP Standards Foundation (früher die Jabber Software Foundation) bei der Entwicklung offener XMPP Erweiterungen aktiv. Der erste IM-Dienst auf Basis von XMPP war Jabber.org, der seit 1999 kontinuierlich betrieben und kostenlose Konten angeboten hat. Von 1999 bis Februar 2006, der Dienst verwendet jabberd als seine Server-Software, zu der Zeit migrierte es zu ejabberd (beide sind kostenlose Software-Anwendungsserver). Im Januar 2010 migrierte der Dienst zur proprietären M-Link Server-Software von Isode Ltd. Im September 2008 erwarb Cisco Systems Jabber, Inc., die Macher des HandelsproduktesJabber XCP. DieXMPP Standards Foundation (XSF) entwickelt und veröffentlicht Erweiterungen zu XMPP durch einen Standardprozess, der auf XMPP Extension Protocols (XEPs, früher als Jabber Enhancement Proposals - JEPs bekannt) ausgerichtet ist. Die folgenden Erweiterungen sind besonders vielseitig einsetzbar: Data Forms Service Discovery Multi-User Chat Publish-Subscribe and Personal Eventing Protocol XHTML-IM Datei Transfer Entity Capabilities HTTPBinding Jingle für Sprach- und Video-Internet der Dinge XMPP Funktionen wie Föderation über Domains, veröffentlichen/subscribe, Authentifizierung und seine Sicherheit auch für mobile Endpunkte werden verwendet, um das Internet der Dinge zu implementieren. Mehrere XMPP Erweiterungen sind Teil der experimentellen Implementierung: Efficient XML Interchange (EXI) Format; Sensor Data; Provisioning; Control; Konzentratoren; Discovery. Diese Bemühungen werden auf einer Seite im XMPP-Wiki für das Internet der Dinge und die XMPP IoT-Mailingliste dokumentiert. Spezifikationen und Standards Die IETF XMPP-Arbeitsgruppe hat eine Reihe von Anfragen für Kommentare (RFC) Dokumente erstellt: RFC 3920 (unterstützt durch RFC 6120) RFC 3921 (unterstützt durch RFC 6121)RFC 3922 RFC 3923 RFC 4622 (unterstützt durch RFC 5122)RFC 4854 RFC 4979 RFC 6122 (unterstützt durch RFC 5122) Die wichtigsten und am weitesten verbreiteten Spezifikationen sind: RFC 6120, Extensible Messaging and Presence Protocol (XMPP:) Core, der Client-Server-Messaging mit zwei offenen XML-Streams beschreibt. XML-Streams bestehen aus <presence,/> <message/> und <iq/> (info/query). Eine Verbindung wird mit Simple Authentication and Security Layer (SASL) authentifiziert und mit Transport Layer Security (TLS) verschlüsselt.RFC 6121, Extensible Messaging and Presence Protocol (XMPP:) Instant Messaging and Presence beschreibt Instant Messaging (IM,) die häufigste Anwendung von XMPP.RFC 7622, Extensible Messaging and Presence Protocol (XMPP:) Address Format beschreibt die Regeln für XMPP-Adressen, auch JabberIDs oder JIDs genannt. Derzeit verwenden JIDs PRECIS (wie in RFC 7564 definiert) zur Verarbeitung von Unicode-Zeichen außerhalb des ASCII-Bereichs. XMPP wurde oft als Konkurrent für SIMPLE angesehen, basierend auf Session Initiation Protocol (SIP) als Standardprotokoll für Instant Messaging und Präsenz-Benachrichtigung. Die XMPP-Erweiterung für Multi-User-Chat kann als Konkurrent mit Internet RelayChat (IRC) gesehen werden, obwohl IRC viel einfacher ist, hat viel weniger Funktionen und ist weit verbreiteter.
