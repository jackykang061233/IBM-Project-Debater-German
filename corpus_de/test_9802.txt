Streaming bezieht sich auf die Liefermethode von Inhalten, anstatt auf den Inhalt selbst. Der Kampf um geistige Eigentumsrechte: A&M Records, Inc. v. Napster,Inc.The lawsuit A&M Records, Inc. v. Napster, Inc. grundlegend verändert die Art und Weise, wie Verbraucher mit Musikstreaming interagieren. Es wurde am 2. Oktober 2000 argumentiert und am 12. Februar 2001 beschlossen. Der Berufungsgerichtshof für den Ninth Circuit entschied, dass ein P2P-Dateifreigabedienst haftbar gemacht werden könnte für ein beitragsmäßiges und kraftvolles Urheberrecht, das als Wahrzeichenentscheidung für das geistige Eigentumsrecht dient. Die erste Frage, die der Gerichtshof angesprochen hat, war "fair use", die besagt, dass ansonsten verletzende Aktivitäten zulässig sind, solange es für Zwecke "wie Kritik, Kommentar, Nachrichtenberichterstattung, Lehre [...] Stipendium oder Forschung ist. Die zweite Behauptung der Klägerin war, dass Napster aktiv zu Urheberrechtsverletzungen beitrug, da sie Kenntnis von weit verbreiteter Aktenteilung auf ihrer Plattform hatte. Dies entspricht etwa 135 GB pro Stunde. Mit einem Multicast-Protokoll sendet der Server nur einen einzigen Stream aus, der allen Benutzern gemeinsam ist. Ein solcher Strom würde daher nur 300 kbit/s zur Verfügung stellen. Weitere Informationen zu diesen Protokollen finden Sie unten. Die Berechnung für Live-Streaming ist ähnlich. Unter der Annahme, dass der Samen am Encoder 500 kbit/s beträgt und wenn die Show 3 Stunden mit 3.000 Zuschauern dauert, wird die Berechnung die Anzahl der übertragenen MB = Encoder-Geschwindigkeit (in Bit/s) × Anzahl der Sekunden × Anzahl der Zuschauer / (8 × 1024 × 1024). Die Ergebnisse dieser Berechnung sind wie folgt: Anzahl der übertragenen MB = 500 x 1024 (bit/s) × 3 × 3.600 (= 3 Stunden) × 3.000 (Anzahl der Zuschauer) / (8 × 1024 × 1024) = 1.977.539MB.In 2018 ist Video mehr als 60% des Datenverkehrs weltweit und macht 80% des Wachstums der Datennutzung aus. Protokolle Der Audio-Stream wird komprimiert, um die Dateigröße mit einem Audio-Codierung Format wie MP3, Vorbis, AAC oder Opus kleiner zu machen. Der Videostream wird mit einem Videocodierformat komprimiert, um die Dateigröße kleiner zu machen. Videocodierungsformate umfassen H.264, HEVC, VP8 oder VP9. Kodierte Audio- und Videostreams werden in einem Container-Bitstream wie MP4, FLV, WebM, ASF oder ISMA montiert. Der Bitstream wird von einem Streaming-Server an einen Streaming-Client (z.B. den Computerbenutzer mit ihrem internetgebundenen Laptop) über ein Transportprotokoll, wie z.B. Adobes RTMP oder RTP, geliefert. In den 2010er Jahren haben sich Technologien wie Apples HLS, Microsofts Smooth Streaming, Adobes HDS und nicht-proprietäre Formate wie MPEG-DASH entwickelt, um adaptive Bitrate Streaming über HTTP als Alternative zu proprietären Transportprotokollen zu ermöglichen. Oft wird ein Streaming-Transportprotokoll verwendet, um Video von einem Veranstaltungsort an einen Cloud-Transcoding-Service und CDN zu senden, der dann HTTP-basierte Transportprotokolle verwendet, um das Video an einzelne Heime und Benutzer zu verteilen. Der Streaming-Client (der Endbenutzer) kann mit dem Streaming-Server über ein Kontrollprotokoll, wie MMS oder RTSP, interagieren. Die Qualität der Interaktion zwischen Servern und Nutzern basiert auf dem Workload des Streaming-Dienstes; da mehr Nutzer versuchen, auf einen Dienst zuzugreifen, wird die Qualität, wenn nicht genügend Bandbreite vorhanden ist oder der Host genug Proxy-Netzwerke nutzt, beeinflusst. Die Bereitstellung von Clustern von Streaming-Servern ist eine solche Methode, bei der es regionale Server gibt, die über das Netzwerk verteilt werden, verwaltet von einem einzigen zentralen Server, der Kopien aller Mediendateien sowie der IP-Adressen der regionalen Server enthält. Dieser zentrale Server verwendet dann Lastausgleichs- und Terminierungsalgorithmen, um Benutzer auf nahe gelegene regionale Server umzuleiten, die sie aufnehmen können. Dieser Ansatz ermöglicht es dem zentralen Server auch, sowohl Benutzern als auch regionalen Servern, die FFMpeg-Bibliotheken verwenden, Daten zu streamen, wenn dies erforderlich ist, so dass der zentrale Server leistungsfähige Datenverarbeitungs- und immense Speicherkapazitäten erfordert. Im Gegenzug sind Workloads auf dem Streaming-Backbone-Netzwerk ausgewogen und gemildert, was eine optimale Streaming-Qualität ermöglicht. Die Gestaltung eines Netzwerkprotokolls zur Unterstützung von Streaming-Medien stellt viele Probleme. Datagram Protokolle, wie das User Datagram Protocol (UDP), senden den Medienstrom als eine Reihe von kleinen Paketen. Dies ist einfach und effizient; es gibt jedoch keinen Mechanismus innerhalb des Protokolls, um die Lieferung zu gewährleisten. Es liegt an der empfangenden Anwendung, um Verlust oder Korruption zu erkennen und Daten mit Fehlerkorrekturtechniken wiederherzustellen. Wenn Daten verloren gehen, kann der Strom einen Ausfall erleiden. Das Echtzeit-Streamingprotokoll (RTSP,) Echtzeit-Transportprotokoll (RTP) und das Echtzeit-Transport Control Protocol (RTCP) wurden speziell entwickelt, um Medien über Netzwerke zu streamen. RTSP läuft über eine Vielzahl von Transportprotokollen, während die beiden letztgenannten auf UDP aufgebaut sind. Ein weiterer Ansatz, der sowohl die Vorteile der Verwendung eines Standard-Web-Protokolls als auch die Fähigkeit zum Streamen sogar von Live-Inhalten zu integrieren scheint, ist adaptive Bitrate Streaming. HTTP adaptive Bitrate Streaming basiert auf HTTP progressivem Download, aber im Gegensatz zum vorherigen Ansatz sind hier die Dateien sehr klein, so dass sie mit dem Streaming von Paketen verglichen werden können, ähnlich wie bei der Verwendung von RTSP und RTP. Zuverlässige Protokolle, wie z.B. das Transmission Control Protocol (TCP), garantieren eine korrekte Auslieferung jedes Bits im Medienstrom. Sie erreichen dies jedoch mit einem System von Timeouts und Retries, das sie komplexer zu implementieren macht. Es bedeutet auch, dass bei Datenverlust im Netzwerk der Medienstrom gestaut wird, während die Protokoll-Handler den Verlust erkennen und die fehlenden Daten erneut übertragen. Clients können diesen Effekt durch Pufferung von Daten für die Anzeige minimieren. Während Verzögerung durch Pufferung in Video-on-Demand-Szenarien akzeptabel ist, werden Benutzer von interaktiven Anwendungen wie Videokonferenzen einen Verlust an Treue erfahren, wenn die Verzögerung durch Pufferung über 200 ms. Unicast-Protokolle senden eine separate Kopie des Medienstreams vom Server an jeden Empfänger. Unicast ist die Norm für die meisten Internet-Verbindungen, aber skaliert nicht gut, wenn viele Benutzer das gleiche Fernsehprogramm gleichzeitig anzeigen möchten. Multicast-Protokolle wurden entwickelt, um die Server-Netzwerk-Lastungen zu reduzieren, die sich aus doppelten Datenströmen ergeben, die auftreten, wenn viele Empfänger unicast Content Streams unabhängig empfangen. Diese Protokolle senden einen einzigen Strom aus der Quelle an eine Gruppe von Empfängern. Je nach Netzinfrastruktur und -typ kann die Multicast-Übertragung möglich sein oder nicht. Ein potenzieller Nachteil der Multicasting ist der Verlust von Video-on-Demand-Funktionalität. Das kontinuierliche Streamen von Radio- oder Fernsehmaterial schließt in der Regel die Fähigkeit des Empfängers aus, die Wiedergabe zu kontrollieren. Dieses Problem kann jedoch durch Elemente wie Cache-Server, digitale Set-Top-Boxen und gepufferte Medienplayer gemildert werden. IP Multicast bietet eine Möglichkeit, einen einzelnen Medienstrom an eine Gruppe von Empfängern auf einem Computernetzwerk zu senden. Ein Multicast-Protokoll, in der Regel Internet Group Management Protocol, wird verwendet, um die Lieferung von Multicast-Streams an die Gruppen von Empfängern auf einem LAN zu verwalten. Eine der Herausforderungen bei der Bereitstellung von IP-Multicast ist, dass Router und Firewalls zwischen LANs den Durchgang von Paketen zu Multicast-Gruppen ermöglichen müssen. Hat die Organisation, die den Inhalt bedient, die Kontrolle über das Netzwerk zwischen Server und Empfängern (d.h. Bildungs-, Regierungs- und Unternehmensintranets), so können Routing-Protokolle wie Protocol Independent Multicast verwendet werden, um Stream-Inhalte in mehrere lokale Netzwerksegmente zu liefern. Wie bei der Massenlieferung von Inhalten benötigen Multicast-Protokolle viel weniger Energie und andere Ressourcen, die weit verbreitete Einführung von zuverlässigen Multicast-Protokollen (broadcast-like) und deren bevorzugte Verwendung, wo immer möglich, ist eine erhebliche ökologische und wirtschaftliche Herausforderung. Peer-to-Peer (P2P)-Protokolle sorgen dafür, dass zwischen Computern vorgezeichnete Streams gesendet werden. Dadurch wird verhindert, dass der Server und seine Netzwerkverbindungen zu einem Engpass werden. Es erhöht jedoch technische, Leistung, Sicherheit, Qualität und geschäftliche Probleme. Aufzeichnungsträger, die livestreamed sind, können über bestimmte Medienplayer wie VLC-Player oder durch die Verwendung eines Bildschirmrecorders aufgezeichnet werden. Live-Streaming-Plattformen wie Twitch können auch ein Video-on-Demand-System enthalten, das eine automatische Aufnahme von Live-Übertragungen ermöglicht, so dass sie später überwacht werden können.
